我们基于 STM32F4+RTOS内核 设计了用于实验室团队自研的水下机器人bricsbot的底层嵌入式系统架构**BRICOS**  在bricsbot中上层香橙派负责高级视觉与导航，本系统负责高频硬实时姿态闭环、传感器采集与硬件驱动等嵌入式功能实现  为了保证各功能的 **可读 可靠 可维护** 我们设计了该嵌入式系统架构

# 系统概述
BRICOS系统是一个复杂实时嵌入式系统  目前是单MCU运行 未来可能会拓展至多MCU 我们最初设计BRICOS的主旨是为了便于移植相关软件工作到不同的MCU和RTOS上 通过BRICOS我们希望能尽量减少底层硬件与上层逻辑的耦合 减少与具体RTOS API的耦合 同时对于后续开发者而言 提供注册-回调接口来便于新算法逻辑的添加

对于机器人系统而言 我们需要保证系统的实时性与各类数据的一致性 提高系统的响应速度 由于水下环境的复杂 系统的各类安全与监测机制也是我们需要考虑的

## 硬件资源清单
+ **执行机构**: 6 x 推进器 (PWM), 1 x 机械臂舵机 (PWM), 2 x 探照灯 (PWM)
+ **传感器组**: 2 x IMU (高频串口), 1 x MS5837 (I2C ), 1 x DHT11 (单总线协议), 1 x 电流电压采样 (ADC)
+ **通信接口**:
+ USART1/2 (高速采集通道): 高频IMU数据采集 (DMA+环形缓冲区)
+ USART3 (RT 通道): 高频硬实时控制与姿态反馈 (DMA + IDLE)
+ UART4 (NRT 通道): 低频管理、调参、OTA 触发 (DMA + IDLE)

# 系统状态设计
系统状态划分 **系统运行模式** 与 **运动控制状态 **两个维度

## 系统运行模式
安全逻辑层 定义了机器人的电源管理与硬件激活级别

**待机模式 (STANDBY)**:

+ **特征**: 低功耗状态
+ **行为**: 推进器完全断电或停发 PWM，停止高频 IMU 解析与 PID 运算。仅保留调试通信 用于上电后进行调试和检测

**工作模式 (ACTIVE)**:

+ **特征**: MCU和传感器正常运作，但电机加锁失能 (Disarmed)
+ **行为**: 无论控制算法输出与否 输出被强行截断，推进器强制输出中位停止信号 (如 1500us)。在此模式下进行传感器校准和下水前检查 (后续考虑添加电子开关直接从电路层面失能电机)

**运动模式 (MOTION - Armed)**:

+ **特征**: MCU 正常工作，电机解锁使能 (Armed)
+ **行为**: 推进器接收经过推力分配矩阵解算后的真实 PWM 值。准备随时执行运动

## 运动控制状态
算法逻辑层 仅当系统处于 **MOTION** 模式时生效，决定 PID 闭环的参与程度

**手动控制 (MANUAL)**:

开环控制。香橙派下发的推力指令未经 PID 处理，直接映射到推力分配矩阵。适用于测试或极端情况自救
此时下发surge sway heave的目标推力值 直接映射到推进器输出 适用于测试或极端情况自救
同时角度下发yaw_cmd 直接映射到偏航角度输出 其他角度不受控制

**定深定向自稳 (STABILIZE)**:

半闭环控制。串级 PID 全面接管。摇杆输入转化为**期望角度**和**期望深度**，由 STM32 内部自动解算补偿推力，抵抗水流干扰
此时下发target_depth 由深度 PID 解算出 heave 推力补偿
下发target_roll=0 target_pitch=0 target_yaw 由角度环和角速度环解算串级PID 输出 推力分配矩阵解算后输出到推进器 闭环控制自身姿态
同时下发surge sway的目标推力值 直接映射到推进器输出 开环控制前进和左右平移

**自主导航 (AUTO)**:

全闭环控制。香橙派下发高级航点或速度矢量，STM32 内部解析并追踪轨迹（后续可拓展路径规划接口）

# 软件架构设计
系统代码整体采用 **纵向三层主链 + System横向公共组件** 的架构思路。  
其中主执行链路自下而上为 **BSP -> Driver -> Task**，而 **System** 不作为线性分层中的“最上层”或“最下层”，而是作为系统运行所依赖的公共基础设施与系统级服务集合，为 Driver 与 Task 提供支撑能力。  
严禁下层直接调用上层 API 接口，向上通知必须通过注册 + 回调机制实现。我们希望通过该架构实现硬件解耦、接口封装与职责清晰。

**BSP 层: ** bsp_uart.c  bsp_i2c.c  bsp_pwm.c

+ 封装ST标准库，提供硬件初始化和回调接口  负责封装硬件相关操作 与芯片直接耦合

**Driver 层：**driver_imu.c driver_ms5837.c

+ 纯逻辑驱动。包含回调分发器、传感器时序协议、校验和计算、推进器执行等

**System 组件**: bot_data_pool.c  FreeRTOS.h

+ 提供 OSAL、任务调度、队列管理、临界区保护、全局数据池等公共能力，保证系统运行与数据一致性

**Task 层**:  task_comm.c  task_rt_cmd.c  task_nrt_cmd.c

+ 具体的业务逻辑封装，如 PID 解算、推力分配矩阵、安全看门狗、任务下发等

## 一、 核心架构组成
BRICOS 的主结构不是简单的线性四层堆叠，而是由一条纵向业务主链和一组横向系统公共组件共同构成。

1. **BSP****层 - L1**
    1. **定位**：底层物理硬件接口封装。
    2. **职责**：封装特定微控制器（MCU）的标准外设外设库，该层与特定硬件平台强耦合，但不包含任何上层协议或业务逻辑。
2. **Driver层 - L2**
    1. **定位**：独立于平台的协议解析与数据处理中枢。
    2. **职责**：与硬件平台及操作系统彻底解耦，调用底层通道的数据，将原始字节流解析为具有实际物理意义的工程量（如姿态角、温度、控制指令）。
3. **Task层 - L3**
    1. **定位**：依业务属性封装的系统执行单元。
    2. **职责**：基于具体的业务需求封装为独立的执行任务（如传感采集任务、姿态闭环控制任务、OTA 通信任务）。
4. **System层 - 横向依赖层**
    1. **定位**：系统运行所依赖的公共能力集合，不参与 BSP -> Driver -> Task 的垂直业务分层排序。
    2. **职责**：为系统提供统一的基础设施与系统级功能组件。包含两大核心组件：
        * **OSAL (****操作系统****抽象层****)**：封装底层 RTOS（如 FreeRTOS）的核心 API（队列、互斥量、任务调度等），提供标准化的系统调用接口，实现应用层代码与特定实时操作系统的解耦。
        * **Bot_Data_Pool (全局数据总线)**：作为所有任务之间状态共享与数据交互的中枢。
    3. **系统功能分支**：
        * **System Core**：OSAL、同步原语、时间基、消息机制、Data Pool 等基础设施。
        * **System Service**：模式管理、监控、故障保护、数据路由等系统级功能模块。
5. **Third_Party**

第三方库。

## 二、 层间与层内交互
#### 1. 层间垂直交互机制
+ **层间调用**：上层模块直接调用下层提供的标准化接口下发指令或请求数据；底层向上层的数据传递及事件通知，必须通过注册回调函数（Callback）实现。
+ **层间数据传递**：底层高速海量数据（如串口 DMA 缓冲）向上传递时采用**指针传递（零拷贝机制）以最小化 CPU 内存开销；当底层解析完毕的数据进入系统全局生命周期（写入全局数据池）时，必须执行值拷贝（深拷贝）**，以确立数据的所有权并保障一致性。

#### 1.1 依赖方向约束
+ **主链方向**：`Task -> Driver -> BSP`
+ **System 使用方式**：System 作为横向公共组件，可被 Driver 与 Task 依赖，但 System 内部不得反向依赖具体 Task 业务模块。
+ **禁止事项**：BSP 不得依赖 Driver/Task 业务逻辑；Driver 不得依赖具体 Task 模块；Task 之间禁止直接形成循环依赖。
+ **向上通知**：Driver 或 BSP 如需向上报告事件，只能通过回调、队列、数据池或事件机制完成，不得直接调用上层业务实现。

#### 2. 层内水平交互机制
+ **水平逻辑解耦**：同层级（特别是 Task 层）的独立任务单元之间，禁止任何形式的直接函数调用或头文件循环依赖。
+ **数据传输**：任务间的状态同步与共享，统一经过全局数据池进行安全读写；任务间的离散事件触发与指令下达，依赖 OSAL 提供的消息队列（Message Queue）或信号量（Semaphore）等中间件实现异步流转。

#### 3.一致性与竞态条件
1. **共享资源的一致性保护**：
    1. 在对全局数据池进行 Push/Pull 操作时，系统依据操作数据的内存块大小及执行耗时，使用**临界区或****互斥锁**进行原子性防护。
    2. **快照访问原则**：在保护期内通过值拷贝创建当前数据的本地副本（Local Copy），随后立即释放锁或退出临界区，使用本地副本执行耗时计算，避免长期独占共享资源引发系统级阻塞。
2. **中断上下文响应约束**：

硬件中断服务例程（ISR）内部绝对禁止调用任何可能导致线程挂起或阻塞的系统 API（如系统级延时、获取常规互斥锁）。中断中调用的回调函数也禁止阻塞。

3. **优先级反转防范**：

如果使用互斥锁就考虑优先级反转问题。

# 内部数据交互机制
由于当前系统具有多线程处理状态、任务和参数信息  设置全局数据池  通过一致性更新和临界区拷贝快照方式保证多线程数据一致性

## 数据结构设计
+ bot_state_t: 存放机器人的物理状态 (姿态角、深度、温湿度、电压)
+ bot_target_t: 存放香橙派下发的期望目标 (目标深度、目标航向、探照灯/舵机指令)
+ bot_params_t: 存放运行模式 (Mode/State)、PID 参数、安全保护阈值等

## API规范
+ 不要在外部文件中直接使用全局变量
+ 读取Pull：任务开始时，使用如Bot_State_Pull(&local_state)将全局数据瞬间快照到任务的局部栈中
+ 写入Push：传感器任务使用局部更新接口(例如 Bot_State_Push_IMU)，底层通过 SYS_ENTER_CRITICAL() 保证结构体写入不会出现读写并发冲突

# 外部数据路由架构
解决高频控制指令与低频数据冲突的问题。

**硬件机制**

UART3 (RT) 和 UART4 (NRT) 均开启 DMA+IDLE。配置双缓冲 (Ping-Pong Buffer) 数组

**零拷贝传递**

中断内不解包，仅通过 FreeRTOS Queue 传递 uint8_t *buf 和 len 至任务层

**数据分发**

+ 两路通道共享同一个可重入的解析引擎
+ 设计函数路由表 根据任务ID进行查找具体处理逻辑函数s_handlers[cmd_id]
+ 上层业务通过 Driver_Protocol_Register(cmd_id, callback_func)  挂载业务逻辑，实现通信底层与业务层的绝对解耦

# 异常安全保护
在复杂水下环境中 保护机器人的物理安全最高优先级

**心跳检测保护**

若 500ms 内未收到香橙派的有效心跳或指令，系统自动触发紧急上浮 (设置期望深度为 0m)

**物理环境报警**

+ **漏水**: 一旦 is_leak_detected 为真，立刻切断推力分配，舵机上锁，闪烁探照灯求救。
+ **超深**: 当实际深度超过 failsafe_max_depth ，拒绝执行任何继续下潜指令。
+ **低压**: 电池电压低于阈值，限制最大输出功率 (max_out)，优先保证通信链路畅通

**硬件看门狗**

看门狗喂狗操作必须在所有核心任务正常运转的前提下进行。发生死锁时强行硬件复位
