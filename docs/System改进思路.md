# System 改进思路（不改代码版）

> 目标：在现有 `System` 横向公共组件的基础上，增加“可扩展、可诊断、可维护”的系统级能力，让下位机更像可复用平台。  
> 原则：不改变 `Task/Driver/BSP` 分层边界，新增能力尽量以“服务/接口”的形式提供。

## 1. 当前 System 现状（基于代码）

现有 `System` 已包含：

- `sys_data_pool`：全局数据池与并发快照
- `sys_log`：异步日志管道（UART4 上报）
- `sys_monitor`：运行时监控（CPU 使用率、芯片温度）
- `sys_systick`：FreeRTOS 兼容 SysTick
- `sys_boot_flag`：OTA 入口标志管理

这些能力已经能支撑基本控制与通信，但还缺少“可调试、可诊断、可扩展”的系统服务层。

## 2. 建议新增的 System 组件方向

### 2.1 交互壳（Shell / CLI）

**目的**：现场调试能力与可观测性快速提升。

- 通过 UART 提供命令行（查看状态、控制参数、触发自检/重启）。
- 支持 `help / status / reboot / set param` 等基础命令。
- 作为通用入口，避免每个 Task 自定义调试指令。

### 2.2 参数服务（Param Service）

**目的**：统一参数注册、默认值、版本迁移与持久化。

建议提供：

- 参数注册表（名称、类型、范围、默认值、读写权限）
- 持久化策略（Flash 写入、版本校验、迁移）
- 统一校验接口（避免非法参数写入）

### 2.3 诊断与自检（Diagnostics）

**目的**：启动自检与在线诊断标准化。

可包括：

- 传感器在线状态（IMU / 深度计 / 电源）
- 执行器健康检查（推进器 PWM 是否可用）
- 通信链路质量（丢包率、超时）
- 自检结果上报（统一格式）

### 2.4 事件总线（Event Bus）

**目的**：减少 Task 间耦合，统一事件传播。

建议：

- 事件发布/订阅（例如 `LEAK_DETECTED`、`LOW_VOLTAGE`）
- 统一事件优先级与去抖逻辑
- System 统一管理告警级别

### 2.5 运行时监控增强（Runtime Monitor+Profiler）

**目的**：扩展现有监控能力，便于性能排查。

建议增加：

- 任务栈水位（High Water Mark）
- 堆内存占用
- 队列长度与溢出统计
- 关键任务运行周期统计

### 2.6 故障管理器（Fault Manager）

**目的**：统一故障等级与恢复策略。

示例：

- `INFO / WARN / CRITICAL` 等级
- 自动降级：如低电压限制推力
- 自动保护：如漏水触发紧急上浮

### 2.7 时间服务（Time Service）

**目的**：统一时间戳与同步上位机时间。

好处：

- 日志与数据对齐更清晰
- 故障定位更容易

### 2.8 OTA 管理器（OTA Manager）

**目的**：让 OTA 流程标准化、可监控。

可增加：

- 升级过程状态机
- 版本检查与回滚策略
- OTA 进度反馈

## 3. 优先级建议（结合 STM32 下位机现状）

### 第一阶段（收益最大）

1. Shell/CLI  
2. 参数服务  
3. 诊断与自检  

### 第二阶段（稳定性提升）

4. 运行时监控增强  
5. 故障管理器  

### 第三阶段（长期维护）

6. 事件总线  
7. 时间服务  
8. OTA 管理器  

## 4. 实施建议（保持分层清晰）

- System 提供公共接口，Task/Driver 只做注册与调用  
- 不直接让 Task/Driver 互相调用  
- 所有新的系统能力通过 System API 统一暴露  

## 5. 后续可选扩展

- 轻量数据库 / KV 存储（配置与运行时数据）
- 录制与回放（用于调试与仿真）
- 多 MCU 协作框架（未来扩展）
