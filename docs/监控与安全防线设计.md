1. 模块设计目标
本模块对应水下机器人（ROV）RTOS 架构中的 Task_Monitor 任务及相关数据结构。在 FreeRTOS 环境下，严禁使用传统的死循环喂狗方式。本模块必须实现“基于独立容忍度的内部任务看门狗”和“基于时间戳的外部通信失控保护”双重防线，确保机器人在发生局部死锁或断连时能够安全自救。

2. 内部防线：工头点名机制 (RTOS Task Health Monitor)
为了防止高优先级任务一直抢占CPU导致低优先级任务饿死，或者某个外设读取死锁，我们采用时间戳打卡机制。

数据结构设计:

在 bot_state_t 中维护一个时间戳数组 uint32_t task_last_tick[MAX_MONITOR_TASKS]，记录每个核心任务最后一次正常运转时的系统 Tick。

独立的超时容忍度 (Timeout Thresholds):

针对不同频率的任务，必须设定不同的超时宽限期。例如：

TASK_ID_CONTROL (100Hz): 容忍度设为 50ms。

TASK_ID_IMU (高频事件驱动): 容忍度设为 100ms。

TASK_ID_COMM_RT (50Hz 通信): 容忍度设为 500ms。

TASK_ID_SENSORS (1Hz 低频采集): 容忍度设为 2000ms。

喂狗逻辑判定:

Task_Monitor 需定期（如 100ms）遍历查岗。计算 当前Tick - 任务最后打卡Tick。

只有当所有任务的未打卡时间都小于各自的容忍度时，才允许调用底层硬件喂狗函数 bsp_wdg_feed()。只要有任何一个任务超时（发生死锁），立刻停止喂狗，等待硬件复位。

3. 外部防线：通信失控保护 (Loss of Signal / LOS Failsafe)
防止伴随计算机（香橙派）死机或通信线缆断裂导致机器人失控猛冲。

时间戳刷新:

每次串口 3 (RT 通道) 成功解析到合法的控制帧时，必须调用 API 更新全局变量 last_comm_tick 为当前系统 Tick。

接管逻辑判定:

Task_Monitor 需计算 当前Tick - last_comm_tick 的差值。

超时阈值: 500ms。

执行动作: 如果差值大于 500ms，且当前系统处于 SYS_MODE_MOTION_ARMED（运动解锁）状态，必须强制调用安全接口，将系统切入 SYS_MODE_ACTIVE_DISARMED（加锁停转）状态。此时底层推力分配矩阵将被强制输出 1500us 中位信号，机器人依靠自身正浮力安全上浮。

5. 与系统模式 FSM 的联动（STANDBY / FAILSAFE）
在引入正式系统模式 FSM 后，监控逻辑需与系统模式强绑定，避免在 STANDBY 期间误判任务超时。

STANDBY 规则：
- STANDBY 下允许传感器任务全部暂停，不应触发“任务超时”故障。
- Task_Monitor 在 STANDBY 期间仅检查自身心跳与通信链路，不检查 IMU/Depth/Power/DHT11。
- 若收到上位机模式切换指令进入 DISARMED，必须在恢复传感器任务后再重新启用完整心跳检查。

FAILSAFE 规则：
- 任意严重故障触发时，Task_Monitor 应推动系统进入 FAILSAFE（而非直接切 DISARMED），由模式管理器统一执行安全动作。
- FAILSAFE 下继续保留通信接收与状态上报，但不再喂狗，直到故障解除并返回 DISARMED。
- 禁止直接从 FAILSAFE 切 ARMED，必须先回到 DISARMED，等待上位机确认。

建议实现细化（设计层）：
- 给 task_last_tick 增加“是否启用监控”的掩码，或由系统模式决定监控范围。
- 在数据池中增加 `fault_flags` 或 `fault_level` 字段，便于上位机获取故障原因。

4. AI 编码强制规范 (Coding Constraints)

临界区保护: Task_Monitor 拉取（Pull）时间戳时，以及其他任务更新（Push / CheckIn）自己的时间戳时，必须使用 SYS_ENTER_CRITICAL() 和 SYS_EXIT_CRITICAL() 进行原子操作保护。

架构解耦: Task_Monitor 只负责判定和修改状态机模式，绝不允许在 Monitor 任务里直接调用底层 PWM 接口停电机。电机的停转由 Task_Control 根据 DISARMED 状态自动执行。
