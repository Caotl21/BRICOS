# 监控与安全防线设计

## 1. 设计目标

BRICOS 的监控与安全防线面向水下机器人运行场景，目标不是简单地“发现异常后打印日志”，而是在上位机失控、传感器异常、任务卡死、低压、漏水等情况下，让 MCU 本地仍然能做出确定、安全、可诊断的响应。

当前安全设计由四层共同组成：

- 任务健康监控：通过任务 check-in 时间戳判断关键任务是否卡死。
- 系统故障监控：采集漏水、IMU、低压、任务超时等故障位。
- 模式安全状态机：由 `System_ModeManager` 统一进入 standby/disarmed/armed/failsafe。
- 可观测与回溯：通过 NRT 状态帧、栈水位帧、UART5 LOG/Shell、持久化日志和故障快照定位问题。

关键原则：

- 安全动作优先级高于上位机命令。
- 监控任务只触发模式和故障状态，不直接绕过控制任务写 PWM。
- 非 armed 模式下，执行机构安全输出由控制任务和驱动层统一落实。
- 故障进入 `FAILSAFE`，恢复必须先回到 `DISARMED`，不能直接回 `ARMED`。

## 2. 当前实现概览

| 项目 | 当前实现 |
| --- | --- |
| 监控任务 | `Task_Monitor` |
| 任务优先级 | `MONITOR_TASK_PRIO = 1` |
| 栈大小 | `MONITOR_STK_SIZE = 256 words` |
| 运行周期 | `MONITOR_PERIOD_MS = 1000`，即 1Hz |
| 硬件看门狗超时 | `bsp_wdg_init(5000)`，约 5s |
| 状态回传通道 | UART4 NRT |
| 日志/调试通道 | UART5 Debug，1500000 baud |
| 故障入口 | `System_ModeManager_EnterFailsafe(fault_flags)` |
| 任务心跳接口 | `Bot_Task_CheckIn_Monitor(task_id)` |
| 任务时间戳读取 | `Bot_Task_LastTick_Pull()` |
| 栈水位回传 | `DATA_TYPE_STATE_STACK_WM 0x07` |

`Task_Monitor` 每秒执行一次，主要流程如下：

```text
读取 CPU/芯片温度
  └─ 更新系统状态数据池
        └─ 拉取系统状态、参数、执行机构、任务 tick、当前模式
              ├─ 收集任务栈水位
              ├─ 判断模式切换保护窗口
              ├─ 判断低压告警
              ├─ 判断任务超时
              ├─ 收集安全故障位
              ├─ 必要时进入 FAILSAFE
              ├─ 健康时喂硬件看门狗
              └─ 通过 NRT 回传系统/执行器/栈水位状态
```

## 3. 任务健康监控

### 3.1 被监控任务

当前 `System/sys_monitor.h` 中定义的任务 ID：

| Task ID | 任务 | 说明 |
| --- | --- | --- |
| `TASK_ID_CONTROL` | 控制任务 | 100Hz 控制环，负责模式执行、PID/TAM 和推进器输出 |
| `TASK_ID_IMU` | IMU 任务 | 姿态/角速度/加速度采集解析 |
| `TASK_ID_MS5837` | 深度任务 | 深度和水温采集 |
| `TASK_ID_POWER` | 电源任务 | 电压、电流采样 |
| `TASK_ID_DHT11` | 舱内环境任务 | 舱内温湿度与辅助状态 |
| `TASK_ID_MONITOR` | 监控任务 | 自身心跳 |

任务通过以下接口打卡：

```c
Bot_Task_CheckIn_Monitor(TASK_ID_xxx);
```

时间戳存放在 `System/sys_monitor.c` 的 `s_task_last_tick[MAX_MONITOR_TASKS]` 中，读写均使用 `SYS_ENTER_CRITICAL()` / `SYS_EXIT_CRITICAL()` 保护。

### 3.2 超时阈值

当前任务超时阈值定义在 `Task/task_monitor.c`：

| 任务 | 超时阈值 |
| --- | ---: |
| `TASK_ID_CONTROL` | 100 ms |
| `TASK_ID_IMU` | 200 ms |
| `TASK_ID_MS5837` | 1000 ms |
| `TASK_ID_POWER` | 10000 ms |
| `TASK_ID_DHT11` | 2000 ms |
| `TASK_ID_MONITOR` | 2000 ms |

如果被监控任务超时：

1. `Task_Monitor` 将 `all_tasks_healthy` 置为 false。
2. 打印 `Task %u timeout` 错误日志。
3. 将 `SYS_FAULT_TASK_TIMEOUT` 合入故障位。
4. 调用 `System_ModeManager_EnterFailsafe()` 进入 `FAILSAFE`。
5. 不执行本轮硬件喂狗，让硬件看门狗作为最后兜底。

## 4. 模式相关监控策略

监控范围不是固定不变的，而是由当前 SysMode 决定。

| SysMode | 当前监控范围 | 原因 |
| --- | --- | --- |
| `STANDBY` | 仅监控 `TASK_ID_MONITOR` | 待机下允许传感器/控制链路低活动或暂停，避免误判 |
| `DISARMED` | 监控全部任务 | 系统已激活但未解锁，需要完整健康检查 |
| `ARMED` | 监控全部任务 | 运动输出使能，必须完整健康检查 |
| `FAILSAFE` | 仅监控 `TASK_ID_MONITOR` | 故障态下保持最小监控闭环，避免故障恢复前反复误判 |

实现入口：

```c
static uint8_t prv_is_task_watch_enabled(bot_sys_mode_e mode, uint8_t task_id)
```

## 5. 模式切换保护窗口

从 `STANDBY` 切换到 `DISARMED` 时，传感器和控制任务可能刚恢复运行，如果立刻检查所有任务心跳，容易误判任务超时。

当前实现增加了模式切换保护窗口：

| 参数 | 当前值 | 说明 |
| --- | ---: | --- |
| `MODE_SWITCH_GRACE_MS` | 800 ms | 最小保护时间 |
| `MODE_SWITCH_GUARD_MAX_MS` | 3000 ms | 最大保护时间 |

保护窗口触发条件：

- 上一模式是 `SYS_MODE_STANDBY`。
- 当前模式切到 `SYS_MODE_ACTIVE_DISARMED`。

保护窗口释放条件：

- 至少经过 800ms。
- 控制任务在模式切换后至少打卡一次。

保护窗口兜底：

- 如果超过 3000ms 仍未满足释放条件，强制释放并打印 warning。

保护窗口期间，监控任务只依据 Monitor 自身健康度喂狗，避免刚恢复任务时出现误报。

## 6. 故障位设计

当前故障位定义在 `System/sys_mode_manager.h`：

| 故障位 | 值 | 当前来源 |
| --- | ---: | --- |
| `SYS_FAULT_LEAK` | `1 << 0` | `bot_sys_state_t.is_leak_detected` |
| `SYS_FAULT_IMU` | `1 << 1` | `bot_sys_state_t.is_imu_error`，以及传感器任务主动触发 |
| `SYS_FAULT_LOW_VOLTAGE` | `1 << 2` | 电池电压低于 `failsafe_low_voltage` |
| `SYS_FAULT_TASK_TIMEOUT` | `1 << 3` | 任务心跳超时 |
| `SYS_FAULT_LOS` | `1 << 4` | 已预留；当前可由调试入口触发，自动 RT 失联检测尚未接入 Monitor 主循环 |

`Task_Monitor` 当前自动收集的安全故障：

```text
leak
IMU error
low voltage
task timeout
```

低压判断有两个口径：

- 告警日志阈值：`MONITOR_LOW_VOLTAGE_WARN_V = 22.0f`，用于打印 `Voltage alarm`。
- failsafe 阈值：`params.failsafe_low_voltage`，用于置 `SYS_FAULT_LOW_VOLTAGE`。

代码中对 failsafe 低压判断增加了 `bat_voltage_v > 1.0f` 条件，避免上电早期 ADC 尚未有效时把 `0.00V` 直接当成真实低压故障。

## 7. FAILSAFE 联动

当 `Task_Monitor` 收集到任意安全故障位时：

```c
System_ModeManager_EnterFailsafe(failsafe_faults);
```

ModeManager 行为：

- OR 累加故障位到内部 `s_fault_flags`。
- 将 `s_sys_mode` 切换为 `SYS_MODE_FAILSAFE`。
- 外部不能通过普通 request 直接进入 failsafe，必须走故障入口。
- failsafe 状态下只允许尝试恢复到 `ACTIVE_DISARMED`。
- 恢复到 `ACTIVE_DISARMED` 前，动态故障必须清零。
- 禁止从 `FAILSAFE` 直接进入 `ARMED`。

控制任务在看到 `SYS_MODE_FAILSAFE` 后进入 failsafe 分支，执行安全输出策略。Monitor 不直接控制推进器 PWM，这是为了避免多个任务同时写执行机构。

## 8. 看门狗策略

当前看门狗采用“任务健康闭环喂狗”：

```text
所有当前应监控任务健康
  └─ Task_Monitor 喂硬件看门狗
否则
  └─ 停止喂狗，等待硬件复位兜底
```

关键点：

- 硬件看门狗初始化超时约 5s。
- `Task_Monitor` 每 1s 检查一次。
- 只有 `all_tasks_healthy == true` 时才调用 `bsp_wdg_feed()`。
- 如果任务超时，系统先进入 `FAILSAFE`，同时不喂狗。
- 如果系统继续无法恢复，硬件看门狗最终复位 MCU。

这是一条最后防线：`FAILSAFE` 用于安全停机，硬件 WDG 用于从软件死锁中恢复。

## 9. 状态回传与可观测性

`Task_Monitor` 每周期通过 UART4 NRT 回传三类状态帧：

| CmdId | 名称 | 内容 |
| --- | --- | --- |
| `0x03` | `DATA_TYPE_STATE_SYS` | SysMode、MotionMode、水温、舱温、湿度、电压、电流、芯片温度、CPU、漏水、IMU、低压 |
| `0x04` | `DATA_TYPE_STATE_ACTUATOR` | 舵机角度、探照灯 1/2 PWM |
| `0x07` | `DATA_TYPE_STATE_STACK_WM` | Monitor、Control、RT Comm、NRT Comm、IMU、MS5837、Power、DHT11、Log 任务栈水位 |

日志输出：

- UART5 Debug 输出实时日志。
- `LOG_WARNING` 和 `LOG_ERROR` 会进入 Flash 持久化日志。
- 重启 replay 的持久化日志带 `[PERSIST]` 前缀。
- NRT 提供 `System_Log_PersistClear()` 对应的带 ACK 清理服务。

启动诊断：

- 启动时会打印复位标志。
- 如果上次发生任务栈溢出，会通过 `System_FaultSnapshot_LoadLastStackOverflowTask()` 读取并打印任务名。
- FreeRTOS 栈溢出 hook 会调用 `System_FaultSnapshot_SaveStackOverflowTask()` 保存快照。

## 10. RT 失联保护现状

`SYS_FAULT_LOS` 已经在故障位中定义，Shell 中 `sysmode set failsafe` 当前会以 `SYS_FAULT_LOS` 作为故障位进入 failsafe，适合人工验证 failsafe 链路。

但当前 `Task_Monitor` 主循环尚未实现自动 RT 失联判断，也没有在合法 RT 控制帧解析处统一更新时间戳。因此文档不能把“500ms 未收到 RT 控制帧自动 failsafe”描述为已完成能力。

建议后续补齐：

1. 在 RT 控制帧成功解析并通过校验后，调用类似 `System_Link_CheckIn_RT()`。
2. Monitor 拉取 `last_rt_tick`。
3. 仅在 `ARMED` 或需要上位机闭环的 motion mode 下启用 LOS 判断。
4. 超时后置 `SYS_FAULT_LOS` 并进入 `FAILSAFE`。
5. 将 LOS 状态回传到 NRT 系统状态或扩展故障状态帧。

建议默认阈值可从 500ms 起步，但应根据上位机 RT 发送周期、串口负载和水下任务策略再确定。

## 11. 数据一致性约束

监控链路涉及多个任务共享状态，必须遵守以下规则：

- 任务心跳时间戳只能通过 `Bot_Task_CheckIn_Monitor()` 更新。
- Monitor 只能通过 `Bot_Task_LastTick_Pull()` 拉取时间戳快照。
- 数据池状态只能通过 `Bot_*_Pull()` / `Bot_*_Push()` 接口访问。
- ModeManager 状态只能通过 `System_ModeManager_*()` 接口切换或读取。
- Monitor 不直接写推进器、舵机、灯光 PWM。
- 状态回传通过 HydroCore 协议帧，不直接暴露内部结构体内存布局。

## 12. 调试与验收建议

建议验证项目：

| 项目 | 预期 |
| --- | --- |
| `STANDBY` 下传感器任务暂停 | 不触发任务超时 |
| `STANDBY -> DISARMED` | 进入保护窗口，等待控制任务打卡后释放 |
| 控制任务卡死 | 触发 `SYS_FAULT_TASK_TIMEOUT`，进入 `FAILSAFE`，停止喂狗 |
| IMU 异常 | 触发 `SYS_FAULT_IMU`，进入 `FAILSAFE` 或阻止 STABILIZE/AUTO |
| 漏水标志为真 | 触发 `SYS_FAULT_LEAK`，进入 `FAILSAFE` |
| 电压低于 failsafe 阈值 | 触发 `SYS_FAULT_LOW_VOLTAGE`，进入 `FAILSAFE` |
| UART4 NRT 状态帧 | 能看到 `STATE_SYS`、`STATE_ACTUATOR`、`STATE_STACK_WM` |
| UART5 Debug | 能看到实时日志和 `[PERSIST]` replay |
| 栈溢出 hook | 下次启动能读取上次溢出任务名 |

当前重点风险：

- `Power` 任务超时阈值为 10s，若电源采样周期调整，需要同步评估该阈值。
- `LOG_INFO("All tasks healthy!!")` 每秒打印一次，现场日志量较大时可能影响可读性。
- `SYS_FAULT_LOS` 尚未自动接入 RT 链路检测，不能依赖它完成自动失联保护。
- `FAILSAFE` 下目前只监控 Monitor 自身，适合故障态最小闭环，但恢复策略必须由上位机或人工确认。
