# BRICOS 系统架构设计

## 1. 架构定位

BRICOS 是为水下具身智能原生设计的嵌入式系统架构。它不是一个单纯的电机控制板固件，而是水下机器人智能系统和真实物理硬件之间的实时执行基座。

它的核心目标是：

- 向上为 ROS、任务规划、感知融合、导航控制和智能决策提供丰富、稳定、可组合的应用接口。
- 向下为推进器、舵机、灯光、惯导、深度、电源和安全监控提供确定性强、可恢复、可诊断的实时控制能力。
- 在水下高风险环境中，把智能算法的不确定性隔离在应用层，把硬件安全、模式约束和故障保护固化在 MCU 实时系统中。

BRICOS 的设计思想可以概括为一句话：

> 上层智能负责“想做什么”，BRICOS 负责“能不能做、如何安全地做、做到什么状态”。

## 2. 系统边界

### 2.1 上位机与智能侧

上位机通常运行 ROS/ROS2、视觉感知、SLAM、任务规划、路径规划、智能体决策、数据记录和人机交互。它通过 BRICOS 提供的 RT/NRT/Debug 通道访问机器人底层能力。

上位机侧主要职责：

- 生成运动目标、姿态/深度目标或任务级控制指令。
- 订阅机器人姿态、深度、电源、环境、任务健康度等状态。
- 下发 PID、TAM、灯光、舵机、安全阈值等非实时配置。
- 执行任务规划、感知融合、语义决策和高层行为控制。

### 2.2 BRICOS 嵌入式侧

BRICOS 运行在 STM32 + FreeRTOS 之上，负责硬实时控制、传感器采集、执行机构驱动、系统模式管理、通信协议、故障检测和安全保护。

BRICOS 侧主要职责：

- 建立统一的硬件抽象和驱动接口。
- 维护全局数据池，为控制、通信和监控提供一致快照。
- 执行 100Hz 控制环、推进力分配、执行机构输出和安全限幅。
- 管理 standby/disarmed/armed/failsafe 系统模式。
- 提供 RT/NRT/Debug 三类通信通道，隔离实时控制、配置服务和日志调试。
- 在故障、失联、低压、漏水、IMU 异常、任务超时等情况下进入安全状态。

## 3. 硬件资源

### 3.1 执行机构

| 资源 | 数量 | 说明 |
| --- | ---: | --- |
| 推进器 | 6 | 主运动执行机构，通过 PWM 输出控制，参与 TAM 推力分配 |
| 舵机 | 2 | 低速姿态/机构类执行机构，通过 PWM 输出控制 |
| 探照灯 | 2 | 高功率照明输出，通过 PWM 调光 |
| WS2812 彩灯 | 2 路 | RGB 状态/交互灯带，通过定时器 PWM + DMA 发送 |

| --- | --- | --- |
| IMU1 | UART1 | 主姿态/角速度/加速度数据源 |
| IMU2 | UART2 | 冗余或扩展姿态数据源 |
| IMU3 | UART6 | 第三路 IMU，面向多惯导冗余和具身状态估计扩展 |

| 电源检测 | ADC | 电池电压、电流 |
| MCU 内部温度 | ADC | 芯片温度与系统健康监测 |
| 漏水检测 | GPIO/传感输入 | 水密舱安全检测 |

| UART | 角色 | 说明 |
| --- | --- | --- |
| UART1 | IMU1 | 第一惯导通道 |
| UART2 | IMU2 | 第二惯导通道 |
| UART3 | RT 通道 | 上位机实时控制与高频状态回传 |
| UART4 | NRT 通道 | 非实时配置、参数、服务、ACK、OTA 等 |
| UART5 | Debug 通道 | LOG 与 Shell 调试通道，当前波特率 1500000 |
| UART6 | IMU3 | 第三惯导通道 |

RT/NRT/Debug 三通道物理隔离后，实时控制不会被日志输出或调试交互阻塞，NRT 服务也不会挤占 RT 控制链路。

## 4. 系统模式设计

BRICOS 使用系统模式管理器作为安全状态机。上层智能不能直接绕过模式约束驱动硬件，所有运动输出都必须经过系统模式、运动模式和故障状态共同裁决。

| SysMode | 含义 | 典型输出策略 |
| --- | --- | --- |
| standby | 待机态，系统已启动但不接受运动输出 | 推进器保持安全输出，控制任务低频运行 |
| disarmed | 解锁前工作态，可通信、可配置、可读取传感器 | 执行机构保持安全，允许准备参数和目标 |
| armed | 运动使能态，允许控制任务驱动推进器 | 根据 motion mode 执行 MANUAL/STABILIZE/AUTO |
| failsafe | 故障保护态 | 强制安全输出，禁止继续执行危险动作 |

当前运动模式包括：

- MANUAL：上位机直接给出运动通道目标。
- STABILIZE：BRICOS 根据姿态/深度目标进行闭环稳定。
- AUTO：面向上层任务规划的自动控制入口。

模式切换原则：

- standby、disarmed、armed 之间只能按安全路径切换。
- armed 进入前必须满足动态故障清除、传感器健康、通信链路健康等条件。
- failsafe 不能由外部命令直接请求，只能由系统故障保护触发。
- failsafe 恢复时必须先回到 disarmed，由上位机或操作员重新确认后再 armed。
- 当 IMU 故障存在时，STABILIZE/AUTO 类依赖姿态闭环的运动模式会被阻止。

## 5. 软件总体架构

BRICOS 采用“分层架构 + 系统组件”的方式组织代码。分层保证依赖方向清晰，系统组件负责跨层通用能力。

```text
┌──────────────────────────────────────────────┐
│ 上位机 / ROS / 具身智能应用                  │
│ 任务规划、感知融合、导航、智能决策、人机交互 │
└─────────────────────┬────────────────────────┘
                      │ RT / NRT / Debug
┌─────────────────────▼────────────────────────┐
│ Task 层                                       │
│ 通信、传感器、控制、监控、NRT 服务、Shell    │
├──────────────────────────────────────────────┤
│ System 组件层                                 │
│ 数据池、模式管理、日志、监控、Shell、PID     │
├──────────────────────────────────────────────┤
│ Driver 层                                     │
│ IMU、MS5837、电源、推进器、舵机、灯光、协议  │
├──────────────────────────────────────────────┤
│ BSP 层                                        │
│ UART、DMA、TIM/PWM、ADC、I2C、GPIO、Flash    │
├──────────────────────────────────────────────┤
│ STM32 + FreeRTOS + 机器人硬件                 │
└──────────────────────────────────────────────┘
```

依赖规则：

- 上层只能通过下层公开接口访问能力，避免跨层直接操作硬件寄存器。
- Task 层通过 System 数据池交换状态，避免任务之间共享可变全局变量。
- Driver 层只描述设备行为，不承载系统模式和业务决策。
- BSP 层只负责 MCU 外设资源映射和基础收发，不理解机器人语义。
- 安全保护优先级高于控制输出，上位机命令永远不能覆盖本地 failsafe。

## 6. 分层设计

### 6.1 BSP 层

BSP 层负责把 STM32 外设资源抽象为稳定接口：

- UART + DMA + IDLE：支持 IMU、RT、NRT、Debug 多通道收发。
- TIM/PWM：支持推进器、舵机、探照灯、WS2812 波形输出。
- ADC：支持电池电压、电流、芯片温度采样。
- I2C：支持 MS5837 等总线传感器。
- GPIO：支持 DHT11、漏水检测、片选、使能等离散信号。
- Flash：支持参数持久化、日志持久化、故障快照和 OTA 标记。
- Watchdog：支持系统健康闭环喂狗。

BSP 是硬件资源的唯一入口。后续更换 MCU 引脚或外设实例时，应优先修改 BSP 映射，尽量不影响 Driver、Task 和 System 层。

### 6.2 Driver 层

Driver 层把具体设备封装成机器人可理解的能力：

- `driver_hydrocore`：BRICOS 通信协议封包、解析、ACK 与数据类型定义。
- `driver_imu`：IMU 数据解析，输出姿态、角速度、加速度等身体状态。
- `driver_ms5837`：深度与水温采集。
- `driver_power`：电源电压、电流与低压判断输入。
- `driver_thruster`：推进器 PWM 输出与安全停止。
- `driver_servo`：舵机角度输出。
- `driver_searchlight`：探照灯亮度控制。
- `driver_ws2812`：双路 WS2812 彩灯 DMA 驱动与发送状态管理。
- 参数/Flash 相关驱动：PID、TAM、安全阈值等配置持久化。

Driver 层应保持“设备驱动”语义，不直接决定机器人能否 armed，也不直接处理上位机业务协议。

### 6.3 System 组件层

System 层提供跨任务、跨驱动的通用系统能力：

- 数据池：统一维护身体状态、系统状态、目标指令、参数、执行机构状态和任务水位线。
- 模式管理：维护 SysMode、MotionMode、故障位和安全切换规则。
- 日志系统：异步日志队列、Debug 通道输出、持久化日志 replay。
- 监控系统：任务心跳、CPU 使用率、任务栈水位、超时检测。
- PID/TAM：控制器参数、推力分配矩阵和控制计算基础设施。
- Shell Core：调试命令注册、解析和执行入口。
- 端口适配：FreeRTOS 临界区、延时、tick、互斥等平台能力封装。
- 故障快照：保存上次复位、任务栈溢出、关键故障信息，便于现场回溯。

System 组件是 BRICOS 的“内脏层”：它不直接面对上位机，也不直接绑定某个硬件，但决定系统是否稳定、可诊断、可恢复。

### 6.4 Task 层

Task 层把系统能力组织成 FreeRTOS 运行实体。每个任务负责一个明确周期或事件流，任务间通过队列、信号量、数据池和协议帧协作。

| 任务 | 优先级 | 周期/触发 | 主要职责 |
| --- | ---: | --- | --- |
| Task_Control | 5 | armed 下 100Hz，standby 下 2Hz | 系统模式裁决、目标读取、PID/TAM、推进器输出、状态回传 |
| Task_IMU | 4 | 周期采样/解析 | 读取 IMU 数据并更新身体状态 |
| Task_RT_Comm | 4 | UART3 DMA/队列事件 | 处理实时控制帧和高频状态链路 |
| Log_Task | 3 | 队列事件 | 异步输出日志，避免业务任务阻塞在串口打印 |
| Task_NRT_Comm | 3 | UART4 DMA/队列事件 | 处理参数、服务、OTA、ACK 等非实时请求 |
| Task_MS5837 | 3 | 周期采样 | 深度和水温更新 |
| Shell RX | 3 | Debug UART 接收事件 | Shell 字节流输入与命令解析入口 |
| Task_Power | 2 | 周期采样 | 电池电压、电流、芯片温度更新 |
| Task_DHT11 | 2 | 周期采样 | 舱内温湿度、漏水辅助状态更新 |
| Task_Monitor | 1 | 1Hz | 健康监控、任务超时、低压/漏水/IMU 故障、栈水位回传 |

任务设计原则：

- 高频控制任务只做控制闭环必需工作，不直接执行慢速 Flash 或日志操作。
- 通信任务只负责收发、解析和分发，业务动作交给 RT/NRT 命令处理模块。
- 监控任务是安全闭环的一部分，负责把任务健康状态转换为系统故障位。
- 日志任务异步化，避免串口打印反向阻塞控制环。
- 每个被监控任务都需要周期性 check-in，监控任务根据超时阈值触发保护。

## 7. 数据设计

BRICOS 使用全局数据池作为任务间共享状态中心。数据池不是“随便读写的全局变量”，而是带有 Pull/Push 边界的快照系统。

### 7.1 核心数据对象

| 数据对象 | 主要内容 | 典型写入者 | 典型读取者 |
| --- | --- | --- | --- |
| `bot_body_state_t` | 四元数、角速度、线速度、加速度、深度 | IMU、MS5837 | 控制任务、RT 回传 |
| `bot_sys_state_t` | 水温、舱温、湿度、电压、电流、芯片温度、CPU、漏水、IMU 状态 | 传感器任务、监控任务 | 监控任务、NRT/RT 回传 |
| `bot_target_t` | 上位机下发的运动目标和 motion mode 相关载荷 | RT/NRT 命令 | 控制任务 |
| `bot_params_t` | PID、TAM、failsafe 深度/低压阈值 | NRT 参数服务、Flash 加载 | 控制任务、监控任务 |
| `bot_actuator_state_t` | 舵机、探照灯等执行机构状态 | NRT 命令、驱动反馈 | 状态回传、Shell |
| `bot_stack_watermark_t` | 各任务栈水位线 | 监控任务 | NRT/RT 状态回传、Debug |

### 7.2 数据流

典型闭环数据流如下：

```text
传感器任务
  └─ Push 身体状态/系统状态
        └─ 数据池快照
              ├─ 控制任务 Pull 目标/状态/参数
              │     └─ PID/TAM/安全裁决
              │           └─ Driver 输出推进器/舵机/灯光
              └─ 通信任务 Pull 状态
                    └─ RT/NRT 回传给上位机
```

数据设计约束：

- 写入数据池必须通过 Push 接口。
- 读取数据池必须通过 Pull 接口获取局部快照。
- 控制任务内部使用本周期快照计算，避免计算过程中状态被异步修改。
- 参数更新要通过 NRT 服务或持久化加载进入数据池，不能绕过系统接口直接改控制器内部状态。
- 状态回传使用协议帧，不把 C 结构体内存布局直接暴露为长期 ABI。

## 8. 通信架构

BRICOS 通信按实时性和用途拆成三类通道。

### 8.1 RT 通道：UART3

RT 通道用于实时运动控制和高频状态交换。它应保持短帧、低延迟、少阻塞。

典型数据：

- 上位机运动目标。
- SysMode/MotionMode 请求。
- 机器人身体状态、系统状态、推进器状态。
- 控制链路必要的实时 ACK 或错误反馈。

### 8.2 NRT 通道：UART4

NRT 通道用于非实时服务，不应该影响控制环稳定性。

典型数据：

- PID 参数读写。
- TAM 矩阵配置。
- 舵机、灯光、WS2812 等应用控制。
- Flash 参数保存/读取。
- OTA 数据和启动标志。
- 持久化日志清理服务 `System_Log_PersistClear()`。
- 带 ACK 的服务调用结果。

### 8.3 Debug 通道：UART5

Debug 通道承载 LOG 与 Shell。当前波特率为 1500000。

典型用途：

- 启动日志。
- 错误、告警、状态诊断。
- `[PERSIST]` 持久化日志 replay。
- Shell 命令调试。
- 现场故障定位和人工操作。

Debug 通道与 NRT 通道分离后，日志脚本不再参与 NRT 协议链路，NRT 客户端只需要关注服务帧和 ACK。

## 9. 控制架构

控制链路以 SysMode 为安全外壳，以 MotionMode 为控制策略，以 PID/TAM 为执行计算核心。

```text
ROS / 上位机
  └─ RT/NRT 目标命令
        └─ bot_target_t
              └─ Task_Control
                    ├─ SysMode 安全裁决
                    ├─ MotionMode 控制策略
                    ├─ PID 姿态/深度闭环
                    ├─ TAM 6 自由度推力分配
                    └─ 6 路推进器 PWM 输出
```

不同模式下的输出策略：

- standby：控制任务低频运行，周期性维护安全输出和 ESC 状态，不执行运动闭环。
- disarmed：允许通信、参数配置、传感器更新，但推进器保持安全。
- armed + MANUAL：执行上位机给出的手动运动目标，并通过安全限幅输出。
- armed + STABILIZE：执行姿态/深度闭环控制。
- armed + AUTO：为任务规划和自主行为预留自动控制入口。
- failsafe：立即压制危险输出，进入故障保护策略。

推进器设计当前面向 6 路实际硬件，TAM 数据结构保留最大 8 路扩展能力，便于未来平台升级。

## 10. 安全保护设计

水下系统的安全保护必须本地闭环，不能依赖上位机“来得及发命令”。BRICOS 将安全保护分成模式保护、故障检测、任务健康、执行机构保护和故障可诊断五类。

### 10.1 模式保护

- 所有运动输出必须经过 SysMode 检查。
- 非 armed 模式下推进器保持安全输出。
- failsafe 不能被外部命令直接解除。
- 从 failsafe 恢复必须先回到 disarmed。
- 存在 IMU 故障时阻止依赖姿态闭环的运动模式。

### 10.2 故障检测

系统故障位覆盖：

- 漏水。
- IMU 异常。
- 低电压。
- 任务超时。
- 上位机链路失联。
- 其他可扩展的硬件或软件故障。

监控任务会周期性读取系统状态和任务心跳，把动态故障同步给模式管理器。

### 10.3 任务健康

每个关键任务都通过 `Bot_Task_CheckIn_Monitor()` 打卡。监控任务维护最后打卡 tick，并按不同任务的超时阈值判断是否异常。

被监控对象包括：

- 控制任务。
- RT 通信任务。
- IMU 任务。
- MS5837 任务。
- Power 任务。
- DHT11 任务。
- Monitor 自身。

系统还会采集任务栈水位线，用于发现栈空间不足、递归/局部数组过大等隐患。

### 10.4 执行机构保护

- 推进器输出由控制任务统一管理，避免多个任务同时写 PWM。
- standby/disarmed/failsafe 下统一输出安全值。
- WS2812 使用 DMA 发送状态管理和超时退出，避免初始化或发送链路死等。
- 灯光、舵机等低速执行机构通过 NRT 服务控制，避免进入实时控制路径。

### 10.5 故障可诊断

- 启动时打印复位原因。
- 记录任务栈溢出快照。
- 支持持久化日志 replay，并用 `[PERSIST]` 前缀区分历史日志。
- NRT 提供持久化日志清理服务，方便上位机在确认故障已采集后清理 Flash 日志。
- LOG/Shell 独立在 Debug 通道，避免诊断输出影响 RT/NRT 协议。

## 11. 面向具身智能的应用接口

BRICOS 向上提供的接口不是单个串口命令集合，而是一组面向智能系统的机器人能力抽象。

| 能力 | 接口形式 | 面向上层的意义 |
| --- | --- | --- |
| 运动目标 | RT 控制帧 | 智能体可以连续地下发动作 |
| 姿态/深度/速度状态 | RT 状态帧 | 为控制、导航、状态估计提供身体反馈 |
| 电源/环境/健康状态 | RT/NRT 状态帧 | 为任务规划和安全策略提供约束 |
| 模式切换 | RT/NRT 命令 + ACK | 上层可以请求状态变化，但必须接受本地安全裁决 |
| PID/TAM 参数 | NRT 服务 | 支持不同载体、不同水域、不同任务的控制重配置 |
| 灯光/舵机/彩灯 | NRT 服务 | 支持交互、照明、任务附件和具身表达 |
| OTA/启动标志 | NRT 服务 | 支持远程维护和系统升级 |
| LOG/Shell | Debug 通道 | 支持现场调试、故障回放和人工干预 |

这组接口让上位机看到的不是零散外设，而是一台具有模式、状态、动作、健康度和可维护能力的水下机器人身体。

## 12. 扩展原则

后续扩展新硬件或新能力时，推荐按以下路径进行：

1. 在 BSP 层完成外设资源和引脚映射。
2. 在 Driver 层封装设备能力，提供初始化、读写和安全停止接口。
3. 在 System 数据池中增加必要的状态或参数。
4. 在 Task 层选择合适的实时周期或事件触发方式。
5. 在 RT/NRT/Debug 接口中暴露给上位机。
6. 在模式管理和安全保护中补齐故障位、限幅和 failsafe 行为。
7. 更新 `docs/ROS上下位机通信接口.md` 和本架构文档，保证代码、协议、文档同步。

对于 IMU3/UART6、多传感器融合、更多推进器、更多灯光通道或更复杂的任务级自主行为，都应遵守同样的分层边界：智能可以越来越强，但实时安全基座必须始终简单、确定、可诊断。
